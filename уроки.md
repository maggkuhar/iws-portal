# Уроки — iws

*Специфические правила, договорённости, причуды этого проекта.*

## Правила

### Порядок разработки блоков — фиксированный, не перескакивать

**Правило:** последовательность строго: Витрина → Магазин → Кабинет → Расписание → Сообщество → Интеграция с ботами → Мобильное приложение.

**Why:** каждый следующий блок опирается на предыдущие. Магазин нужен прежде Кабинета (без товаров — что показывать в истории заказов?). Сообщество требует Кабинета (нужны профили). Интеграция с ботами — на готовый Кабинет. Мобилка — на API всего портала.

**How to apply:** если пользователь предложит «давай сначала Сообщество» — напоминаю про порядок и что готового Кабинета нет. Решение менять — только осознанно, не «потому что хочется».


### БД — PostgreSQL, не SQLite

**Правило:** для IWS используем PostgreSQL (`psycopg2` / `asyncpg`). НЕ SQLite, как в большинстве других проектов сервера.

**Why:** портал = магазин + сообщество + сообщения = много конкурентной записи (одновременные регистрации, заказы, посты). SQLite блокирует БД на запись — будут таймауты при росте.

**How to apply:** при создании нового блока — миграция Postgres (Alembic или ручные SQL). Не использовать `sqlite3` импорт. Тестовая БД — тоже Postgres, не SQLite в памяти.


### Боты IWS не переписываем — только интегрируем

**Правило:** существующие боты `@iwsgamebot` и `@iwsdiarybot` — рабочие продукты с пользователями. Не переписывать, не «улучшать архитектуру». Блок 6 (Интеграция с ботами) — это **синхронизация** (TG-аккаунт ↔ профиль на портале), не переезд функционала.

**Why:** у ботов уже есть пользователи и история. Любая переделка = риск сломать живой продукт. Портал — отдельный канал, не замена.

**How to apply:** если возникнет соблазн «давай вынесем логику бота в портал» — отвергаем. Только связь по userId / token.


### Архитектура — Flask blueprints, как в остальных проектах

**Правило:** один блок = один blueprint = одноимённая кириллическая папка. `app.py` только склейка. Тот же шаблон что в `blogery`/`alexwake`.

**Why:** консистентность. Когда работаем над несколькими проектами — переключение проще.

**How to apply:** структуру нового блока копирую из ближайшего готового (например, `alexwake/галерея/` или `blogery/блогеры/`).


### Концепция зафиксирована 2026-04-06 — не ломать без явного ТЗ

**Правило:** список продуктов бренда (онлайн + офлайн), 7 блоков портала и порядок разработки утверждены пользователем 2026-04-06. Не предлагать переосмыслить структуру / убрать/добавить блоки без явной просьбы.

**Why:** концепция уже была обсуждена и утверждена. Возврат к обсуждению = повтор решённого, времени не добавляет.

**How to apply:** если в чате я начинаю «а давайте Магазин разделим на...» — стоп, это уже зафиксировано. Только реализация.


### Оплата — выбор провайдера ОТЛОЖЕН до Магазина

**Правило:** ЮКасса / Тинькофф / Stripe — список кандидатов, но финальный выбор делаем при разработке Магазина (блок 2), не раньше.

**Why:** выбор зависит от: где будут продаваться товары (РФ / международно), какой объём, какие комиссии готовы платить, какие требования по чекам (54-ФЗ). Эти параметры уточняются ближе к запуску.

**How to apply:** в Витрине (блок 1) интеграцию оплаты не закладываем. В Магазине — сначала обсуждение, потом интеграция.


### Хранилище медиа — выбор отложен до нагрузки

**Правило:** S3 vs локальное хранилище — решаем когда будет реальный объём контента. До тех пор — локально (`uploads/`).

**Why:** S3 даёт CDN и масштаб, но требует $ и ключи. Если медиа мало — локального диска хватит.

**How to apply:** старт — локально. При появлении проблем (диск, скорость) — миграция на S3 как отдельная задача.
